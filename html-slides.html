Chapter 04
Why Protection
Fails
25
The Mismatch
WAFs see packets.
WordPress attacks hide in application logic.
What WAFs were built for
Generic attacks
SQL injection in URL parameters
XSS in form fields
Path traversal in file requests
What WordPress attacks look like
Normal traffic
Valid admin-ajax.php call with malicious payload
Legitimate plugin endpoint exploited through its own logic
Privilege escalation via plugin authentication flow
26
Transparency
Why we didn't
name the hosts
Naming hosts creates false safety — "my host isn't on the list, I'm fine." That's not true.
We tested additional providers privately. Same results. The tools are industry-standard: Cloudflare, Imunify360, ModSecurity.
The finding
Different hosts running the same tools showed the same gaps. This is a structural problem, not a vendor problem.

Chapter 05
The
Solution
30
Defense in Depth
Layer 01 / Network
Your host's
first line of defence
Host WAF, DDoS protection, IP reputation filtering. Handles volumetric attacks and known bad actors well.
But
Blocks ~26% of WordPress-specific attacks. The other 74% pass straight through the holes.
31
Defense in Depth
Layer 02 / Application
WordPress-aware
protection
A WordPress-aware firewall understands plugin request context — it catches what network WAFs can't see.
Virtual patching
Blocks known exploits before a code fix exists — closing the gap between disclosure and patch.
But
Still has holes. Zero-days and unknown plugins can slip through.
32
Defense in Depth
Layer 03 / Operational
Every layer has holes.
Together they don't.
Vulnerability monitoring, fast patching cadence, incident response plan. Shrinks the window between disclosure and defence.
No single layer blocks everything. The Swiss cheese model works because the holes never align across all three.
